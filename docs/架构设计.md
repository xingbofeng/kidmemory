# 0. 整体架构结论

KidMemory 后续架构分为四层：

```txt
Web / Vercel
  - 上传页
  - 分享页
  - 落地页
  - 轻量管理页

Cloud API / 腾讯云
  - 上传
  - 分享
  - 设备同步
  - 轻量任务状态
  - 云端 artifact metadata

Local Sidecar / 本地
  - 本地 DB
  - 本地素材路径
  - 本地搜索 / embedding
  - 本地 Agent
  - workspace
  - PDF / 长图导出

Desktop / Flutter
  - 本地 UI
  - 启动 sidecar
  - 调用 sidecar
  - 设备绑定
```

当前原则：

```txt
1. Agent 继续本地做，不拆 agent-runtime。
2. cloud-api 不跑 Agent，只做上传、分享、设备同步、轻量任务状态。
3. sidecar 继续管本地 DB、本地素材、本地 Agent、本地导出。
4. Web 继续部署在 Vercel。
5. cloud-api 部署在腾讯云，入口走 Cloudflare Tunnel。
6. sidecar 不部署到云端，后续随 Desktop 打包。
7. 所有 JSON API 统一返回 code/msg/data。
8. 协议统一放 packages/protocol。
9. cloud-api 和 sidecar 使用同一套技术栈和工程结构。
10. 不引 Redis，不引额外限流组件，防刷先用 PostgreSQL。
```

# 1. 拆单原则

```txt
1. 一个单解决一个完整工程风险，不按文件拆。
2. 一个单必须能独立验收。
3. 协议、接口、客户端、测试要一起闭环。
4. 安全相关必须有并发测试和 contract tests。
5. cloud-api 和 sidecar 是两个服务，不是两套技术栈。
6. Web 和 Desktop 都不能直接散落 API 请求。
7. 生产代码不能混 mock/sample/demo fallback。
8. CI 必须覆盖 protocol / sidecar / cloud-api / web / desktop。
9. 桌面 release 先做 dry-run，不做正式签名公证。
```

# Epic 1：协议层与统一响应协议

## 1. 背景

当前项目缺少统一协议层。后端、Web、Desktop 之间存在以下问题：

```txt
1. 错误码散落在模块内部。
2. 响应格式不统一。
3. Web 端手写 axios 请求。
4. Desktop 端手写 Map 解析。
5. 后续 cloud-api / sidecar 拆分后，协议可能进一步分裂。
6. i18n 需要稳定错误码作为翻译 key。
7. OpenAPI / Proto / 生成代码需要统一出口。
```

因此需要新增 `packages/protocol`，作为整个项目的协议唯一真相源。

## 2. 用户故事

作为开发者，  
我希望所有 API 响应结构、错误码、DTO、OpenAPI、Proto 都在一个地方维护，  
以便 Web、Desktop、sidecar、cloud-api 不再重复手写协议。

作为前端开发者，  
我希望可以从协议层生成 TypeScript client，  
以便页面不再手写 axios URL 和响应解析。

作为桌面端开发者，  
我希望可以从协议层生成 Dart 类型，  
以便 Flutter 侧不再到处写 `Map<String, dynamic>` 解析。

作为维护者，  
我希望协议变更能被 CI 检查，  
以便避免后端改了接口而客户端没跟上。

# Epic 2：全量接口统一为 code/msg/data

## 1. 背景

当前后端返回格式不统一。

已有情况：

```txt
1. 部分错误通过 GlobalExceptionFilter 统一成 ok/code/message/timestamp/path。
2. 部分中间件直接返回 error/message。
3. 成功响应完全不统一，例如 { children }、{ asset }、数组、HTML、文件流等。
4. Web 和 Desktop 需要适配多种响应结构。
5. i18n 和错误码处理无法稳定基于 code。
```

需要统一所有 JSON API 返回：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

## 2. 用户故事

作为前端开发者，  
我希望所有 JSON API 都返回统一结构，  
以便前端只写一套解包和错误处理逻辑。

作为桌面端开发者，  
我希望 SidecarApi 可以稳定解析 `code/msg/data`，  
以便失败时不会静默返回空对象。

作为维护者，  
我希望所有错误都有数字 code，  
以便日志、i18n、测试、文档都可以稳定关联。

## 3. 范围

# Epic 3：PG-based 上传 / 分享 / Session 防刷

## 1. 背景

当前已有基础防刷，但主要基于内存 Map：

```txt
RateLimitMiddleware：
- IP 100/min
- Global 1000/min
- 创建 session 20/min

SessionQuotaMiddleware：
- 每个 childId 最多 5 个 active sessions
- 每个 childId 每天最多 20 个 sessions
```

问题：

```txt
1. 重启后状态丢失。
2. 多实例部署时不共享状态。
3. 上传链路只保护了创建 session，没有完整覆盖 create items / commit / pullback。
4. 分享 assets / book / preview 访问需要统一消费访问次数。
5. middleware 当前返回格式不是 code/msg/data。
6. 不希望引入 Redis 或第三方组件。
```

因此需要基于 PostgreSQL 实现统一防刷、配额、幂等和轻量任务领取能力。

## 2. 用户故事

作为使用手机上传照片的家长，  
我希望上传链接不会被别人恶意刷爆，  
以便我的上传和分享体验稳定可靠。

作为维护者，  
我希望所有上传和分享防刷状态保存在 PostgreSQL，  
以便服务重启、多实例部署后限流仍然有效。

作为开发者，  
我希望上传 commit 和 pullback 是幂等的，  
以便重复请求不会造成重复导入、重复下载或数据污染。

## 3. 范围

### 本单包含

- [ ] 新增 `rate_limit_buckets` 表
- [ ] 新增 `PgRateLimitService`
- [ ] 新增 `RateLimitCleanupService`
- [ ] 新增 `UploadSecurityService`
- [ ] 新增 `ShareSecurityService`
- [ ] 创建上传 session 走 PG 限流
- [ ] upload items 创建走 PG 限流
